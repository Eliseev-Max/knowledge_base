Если вы запустите ssh-agent вручную в окне терминала,
(eval `ssh-agent`  OR  eval $(ssh-agent))
функциональность агента будет доступна только в этом окне и процессах, запущенных из него.
Чтобы агент был доступен во всех терминальных окнах сеанса GUI,
(!) необходимо, чтобы он запускался в рамках настройки сеанса GUI.

Ваш дистрибутив может иметь или не иметь некоторые средства для этого:
  -> дополнительный PAM-модуль pam_ssh.so может сделать это,
  -> или Debian и родственные дистрибутивы могут включить SSH-агент для сессий X11
     через /etc/X11/Xsession.options (см. man 5 Xsession.options для получения дополнительной информации).
  - #В других дистрибутивах могут быть другие решения.

Технически, когда запускается SSH-агент, он создает сокет Unix, который позволяет SSH-клиентам взаимодействовать с агентом, а переменная окружения SSH_AUTH_SOCK используется для определения его местоположения.
При использовании переадресации агента sshd удаленного узла создаст аналогичный сокет и переменную окружения для переадресованного соединения агента.

Если вы
  1) запустите агента в одном локальном окне терминала,
  а затем
  2) скопируете эту переменную окружения ($SSH_AUTH_SOCK) в другой терминал, где также запущена локальная оболочка,
  -> то вы сможете использовать запущенного вручную агента в обоих терминалах.

[->] Но обычно лучше найти решение для запуска агента в рамках настройки сеанса GUI,
     чтобы переменная окружения SSH_AUTH_SOCK была уже настроена и автоматически наследовалась всеми терминалами, запущенными в этом сеансе.

Обратите внимание, что администратор удаленного узла может запретить пересылку агента SSH, если сочтет это необходимым.

В ssh есть два способа входа без пароля:
    * однократный вход с помощью kerberos или AD (которые используют kerberos)
      # не понимаю, при чём здесь kerberos/AD
    * pub-key

  ssh-agent может только "переслать" закрытый ключ, например:
-->
1) Я запускаю ssh-agent:
$ eval `ssh-agent`
Agent pid 26541

2) Добавляю ключ
#[!]Если агент запущен без параметров, то ssh-add добавит файл:
#>  $HOME/.ssh/identity
# Если идентификация содежит ключевую фразу, то ssh-add запросит эту ключевую фразу
# (используя маленькую программу Х11, если запущено из-под Х11, или из терминала, если запущено без Х).

$ ssh-add		# использовать ssh-ключ по умолчанию
|---|
$ ssh-add ~/.ssh/<private_ssh_key>		# использовать конкретный SSH-ключ

3) ..
<--
{}
