# Global Tool Configuration.Dependency-Check.Name = dependency-check-9.2.0

Dependency-Check - это утилита, которая выявляет зависимости проекта и проверяет наличие известных, публично раскрытых уязвимостей.
Этот инструмент может стать частью решения проблемы OWASP Top 10 2017: A9 - Использование компонентов с известными уязвимостями.
Этот плагин может самостоятельно выполнять анализ Dependency-Check и визуализировать результаты.

#################
# Использование #
#################

  Плагин состоит из трех основных компонентов:
    1) глобально определяемой конфигурации инструментов (globally defined tool configuration),
    2) конструктора (builder)
    3) издателя (publisher)

*****************************
* Global Tool Configuration *
*****************************

  Одна или несколько версий Dependency-Check могут быть установлены через Jenkins Global Tool Configuration.
  Установка Dependency-Check может быть выполнена:
    * автоматически
        (при этом будет загружен и извлечен официальный интерфейс командной строки (CLI) с Github)
      <либо>
    * вручную
        (официальный дистрибутив может быть установлен вручную,
         а путь к установке будет указан в конфигурации.)

***********
* Builder *
***********

  Разработчик выполняет анализ, используя одну из предопределенных установок CLI Dependency-Check.
  Конфигурация, специфичная для Jenkins, минимальна:
    важным аспектом конфигурации job'ы является поле 'Arguments',
    которое отправляется непосредственно в определенную установку CLI.

  В версии 9.0.0 dependency-check перешла от использования потока данных NVD к NVD API.

[!] Пользователям dependency-check настоятельно рекомендуется получить NVD API Key;
    (см. https://nvd.nist.gov/developers/request-an-api-key )
    Без NVD API Key обновления dependency-check будут происходить крайне медленно.
    Как установить ключ NVD API, смотрите в документации к интеграциям cli, maven, gradle или ant.

  API NVD имеет принудительные ограничения скорости.

[!] ЕСЛИ
    * вы используете один API KEY
      <и>
    * происходит несколько сборок

[!] => вы можете превысить лимит скорости и получить 403 ошибку.

  В среде CI необходимо использовать:
    -> стратегию кэширования
      или
    -> внешнюю базу данных, обновляемую еженедельно по расписанию.


*************
* Publisher *
*************

  Publisher РАБОТАЕТ НЕЗАВИСИМО от tool configuration или Builder'а
  и отвечает:
    -> за чтение файла dependency-check-report.xml
    -> за генерирование
          * метрик (metrics),
          * тенденций (trends),
          * выводов (findings),
    -> за отказ сборки (Failing)
         или
          перевод ее в состояние предупреждения (Warning State) на основе настраиваемых пороговых значений.

[!] Если в задании настроен Publisher,
    на графике тенденций будет отображаться общее количество обнаруженных фактов, сгруппированных по степени серьезности.

[*] График является интерактивным.
    При наведении курсора на сборку отображается информация о степени серьезности.

  Можно просматривать результаты по КАЖДОЙ сборке.
  Результаты отображаются в интерактивной таблице, которую можно сортировать, осуществлять по ней поиск и листать.
  Каждый результат можно раскрыть, чтобы получить дополнительные сведения.


##########################
# Запрос ключа API (NVD) #
##########################

  Чтобы запросить API-ключ NVD, укажите:
    - название вашей организации,
    - действующий адрес электронной почты
    - и тип вашей организации.

  Для получения ключа API необходимо
    1) прокрутить страницу до конца Соглашения об условиях использования  и
    2) отметить "Я согласен с условиями использования".

  После отправки запроса вы получите электронное письмо, содержащее одноразовую гиперссылку,
  которая используется для активации и просмотра вашего ключа API.

  Если в течение семи дней ключ НЕ БУДЕТ АКТИВИРОВАН,
  необходимо отправить новый запрос на получение ключа API.
